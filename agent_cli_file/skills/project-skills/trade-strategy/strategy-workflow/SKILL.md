# Strategy Workflow Skill

Each strategy family has its own workspace under `strategies/<strategy_family>/`.

## Folder Purpose Reference

| Folder / File | What goes here |
|--------------|---------------|
| `README.md` | Single strategy spec file: scope, hypothesis, spec, implementation notes, open questions. The handoff start point for every agent session. |
| `STATUS.md` | **YAML frontmatter only.** Contains: phase, last_backtest_stem, last_run_metrics, next_action, last_updated. Nothing else. Human narrative → `sessions/<date>-status-note.md`. Use `session-close` CLI to update — do not hand-edit. |
| `hypothesis-log.md` | Ordered list of all tested hypotheses with verdicts. Always check before proposing a new change to avoid circular exploration. |
| `sessions/` | Agent-user conversation summaries, decision context, analysis summaries, iteration plans. Not formal records — captures *why* a decision was made and what was found. |
| `sessions/_latest.json` | Machine-readable pointer to the most recent analysis summary and iteration plan filenames. Update whenever a new session file is written. |
| `_context_digest.md` | Compressed agent-resume artifact: strategy summary, current phase, last metrics, next action, active skill. **Auto-generated by `session-close` CLI** — never hand-edit. Always reflects current STATUS.md + _latest.json state. |
| `experiments/` | Reproducible experiment comparisons: a specific change vs baseline, with concrete metrics and a KEEP/REVERT conclusion. Must have data. |
| `reports/` | Formal backtest summaries: a completed run with key metrics, OOS check, verdict, and next actions. Written after a run is concluded. |
| `reviews/` | Risk logs, spec review notes, and decision records. Written when validating or auditing strategy logic. |
| `engine/freqtrade/` | Engine-specific config. Generated by `generate_freqtrade_config.py`, not hand-filled. |

**Key distinction — sessions vs experiments:**
- `sessions/` = *why* a decision was made, *what was found* (analysis summaries, iteration plans, discussion)
- `experiments/` = *what was tested* and *what the data showed* (reproducible, has numbers, has conclusion)

## Single-File-First Rule

Strategy work starts as a single `strategies/<strategy_family>/README.md` that combines scope, hypothesis, spec, and implementation notes.

**Do not split this file unless the user explicitly asks.** A single file gives agents full context in one read and avoids agents reading only part of the spec.

`STATUS.md`, analysis summaries, and iteration plans in `sessions/` are **supplements** to `README.md` — they provide current state and handoff context. They do not replace any section of `README.md`. When a hypothesis or spec changes, update `README.md` directly.

If the user requests a split, use: `scope.md`, `hypothesis.md`, `spec.md`, `implementation.md`. Preserve the README.md until the user confirms the split is complete.

## Record Placement Rules (Single Source of Truth)

- Handwritten formal backtest reports → `strategies/<family>/reports/`
- Experiment comparisons with data → `strategies/<family>/experiments/`
- Validation, testing, troubleshooting notes → `strategies/<family>/experiments/` (promote to `reports/` only when formally concluded)
- Risk and spec review notes → `strategies/<family>/reviews/`
- Conversation, decision context, analysis summaries, iteration plans → `strategies/<family>/sessions/`
- Raw engine-generated output (HTML, JSON, ZIP) → `freqtrade/reports/` or `freqtrade/user_data/` — **never** in `strategies/`
- Cross-strategy lessons → `knowledge-base/knowledge/backtest-records/`

Do not place records in the repo root, `lib/`, `freqtrade/strategies/`, or any path outside `strategies/<family>/`.

## Phase Gate Checklist

Verify these conditions before moving to the next phase. Do not proceed if gates are not met.

**Implementation → Backtest:**
- [ ] `freqtrade/strategies/<StrategyName>.py` exists with no TODO markers
- [ ] `strategies/<family>/engine/freqtrade/config.json` is generated (not template placeholder)
- [ ] `python -m lib.endpoints.freqtrade` does not raise an import error
- [ ] `STATUS.md` phase updated to `backtest`

**Backtest → Analysis:**
- [ ] `freqtrade/user_data/backtest_results/<stem>.zip` exists
- [ ] **Preferred (full pipeline):** `python -m lib.endpoints.run_backtest_suite <family> <StrategyName> --is-timerange YYYYMMDD-YYYYMMDD [--oos-timerange YYYYMMDD-YYYYMMDD] [--commit]`
  (chains: data check → IS backtest → OOS backtest → IS/OOS reports → session_close → check-gate)
- [ ] **Manual alternative:** `python -m lib.endpoints.session_close <family> --is-stem <stem> [--oos-stem <stem>] --phase analysis`
  (updates STATUS.md, _latest.json, _context_digest.md automatically)

**Analysis → Planning:**
- [ ] Run the full diagnostic checklist from `backtest-analysis-review/SKILL.md` (R:R math → MAE/MFE → regime segmentation → exit analysis → root cause classification)
- [ ] If hypothesis involves a major direction change: run multi-model review (Gemini + Codex + Claude) via `multi-agent-consensus/SKILL.md`; save at `sessions/<YYYYMMDD>-consensus-analysis-<topic>.md`
- [ ] `strategies/<family>/sessions/<YYYYMMDD>-analysis-summary.md` written (must include: R:R math, MAE/MFE interpretation, regime table, root cause type, proposed hypothesis)
- [ ] Run: `python -m lib.endpoints.session_close <family> --is-stem <stem> --phase analysis --analysis-summary <filename>`
- [ ] Ask: does this result surface a cross-strategy pattern? If yes, write to `knowledge-base/knowledge/backtest-records/<YYYYMMDD>-<topic>.md`
- [ ] Run: `python -m lib.endpoints.strategy_cycle check-gate <family>` → must PASS
  (gate verifies: hypothesis-log.md has the IS stem recorded with a non-PENDING verdict)

**Planning → Implementation (next iteration):**
- [ ] Prior art check: `hypothesis-log.md` reviewed for the proposed change
- [ ] `strategies/<family>/sessions/<YYYYMMDD>-iteration-plan.md` written
- [ ] `strategies/<family>/sessions/_latest.json` updated with new iteration_plan filename
- [ ] New hypothesis version added to `hypothesis-log.md` with PENDING verdict
- [ ] `README.md` updated if hypothesis or spec changed
- [ ] Run: `python -m lib.endpoints.session_close <family> --is-stem <stem> --phase planning --iteration-plan <filename>`
- [ ] **Commit all changed files** including `knowledge-base/` if any skills were added or updated this cycle

## What NOT to Do

- Do not create extra review or session files unless the user explicitly requests one.
- Do not mix engine-generated outputs with hand-written summaries in the same folder.
- Do not split README.md automatically when strategy content grows — wait for the user's instruction.
- Do not update STATUS.md speculatively — only update when a phase gate is actually reached.
- Do not add prose body to STATUS.md — it is frontmatter-only. Use `session-close` CLI to update.
- Do not hand-edit `_context_digest.md` — it is auto-generated by `session-close`.
